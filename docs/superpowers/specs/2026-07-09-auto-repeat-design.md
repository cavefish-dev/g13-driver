# Auto-repeat (typematic) for held keys — design

- **Status:** approved (design)
- **Date:** 2026-07-09
- **Scope:** Make held hold-means-hold key bindings **auto-repeat** like a physical keyboard
  (initial delay, then a steady repeat rate), configurable globally (timing) and per-binding
  (on/off). Reuses the existing dispatcher / injector / hot-reload stack.

## Motivation

A held G-key/thumb binding currently injects a single key-down: the key is held (correct for
games, which read held state) but text fields receive exactly **one** character. A physical
keyboard repeats. We want the same.

Windows does **not** auto-repeat injected keys: typematic repeat is generated by the OS keyboard
stack from the *physical device's* held state, at the driver/raw-input layer. `SendInput` injects
above that layer, so a held injected key-down never arms the typematic timer (this is why
AutoHotkey and on-screen keyboards implement their own repeat). Making the OS repeat for us would
require presenting as a virtual HID keyboard device (kernel/signed driver) — a large scope jump
that breaks our user-space `SendInput` design. So we **re-inject** the key at the repeat rate
ourselves. This is the standard, proportionate solution.

## Approach (chosen)

Drive a periodic **tick** from the existing consumer loops into the dispatcher; the dispatcher
re-fires held, repeat-enabled keys on schedule. Timing is app-global (manifest); enable is
per-binding (profile). No new thread, no locks — the dispatcher stays single-threaded and all
repeat state lives inside the existing `held_keys`.

Rejected alternatives:
- **Dedicated repeat thread:** introduces shared mutable `held_keys` behind a lock and concurrency
  with the dispatch thread, for no user-visible benefit over a tick.
- **Virtual HID keyboard device (let the OS repeat):** kernel/signed driver, breaks WinUSB
  user-space design. Out of scope.
- **Inline-table binding values** (`G2 = { key = "w", repeat = true }`): forces `[keys]` values to
  be string-or-table, complicating parsing and breaking the "value is always a combo string"
  contract the GUI and `to_toml` rely on. Rejected in favor of a parallel `[repeat]` table.

## Data flow

```
USB event ──► handle(event) ──┐
                              ├─► Dispatcher (held_keys, each with a repeat schedule)
tick(now) ────────────────────┘        │
   every ~15ms from consumer loop      └─► injector.key_down(combo.key)   ← the repeat
```

## Components

### `src/config.rs` (TDD)

**Global timing (manifest `config.toml`).** New `[autorepeat]` section:

```toml
[autorepeat]
delay_ms = 400      # wait before repeating starts
interval_ms = 40    # gap between repeats (~25/sec)
```

- New struct `AutoRepeat { delay_ms: u64, interval_ms: u64 }`.
- Defaults when the section is absent: `delay_ms = 400`, `interval_ms = 40`.
- Clamp on load: `interval_ms` floored to `1` (a `0` interval would busy-spin); `delay_ms` may be
  `0` (repeat immediately after first tick).
- `ProfileSet` gains the value + accessor `autorepeat() -> AutoRepeat`, loaded from the manifest
  (and from a legacy inline single-profile `config.toml`, which simply yields defaults if the
  section is absent).

**Per-binding enable (profile file).** A parallel `[repeat]` table alongside `[keys]`:

```toml
[keys]
G1 = "ctrl+c"
G2 = "w"

[repeat]
G2 = true          # only keys listed true repeat; default off
```

- `Profile` gains `repeat: HashMap<G13Key, bool>`, parsed from `[repeat]` using the existing
  `parse_g13_key` (so `G1`/`BTN1`/`STICK`/… all work, case-insensitive).
- Accessor `repeats(&self, key: G13Key) -> bool`, default `false` (a key absent from `[repeat]`,
  or present and `false`, does not repeat).
- `to_toml` emits a `[repeat]` section listing only the enabled keys (Debug-name form, same
  convention as `[keys]`), so a save/reload round-trip preserves the flags. Disabled keys are
  omitted, not written `false`.

Both changes are additive and backward-compatible: an old profile with no `[repeat]` and a
manifest with no `[autorepeat]` behave exactly as today.

### `src/dispatcher.rs` (TDD)

`held_keys` changes from `HashMap<G13Key, KeyCombo>` to `HashMap<G13Key, HeldKey>`:

```rust
struct HeldKey {
    combo: KeyCombo,
    repeat: bool,                 // snapshot of profile.repeats(key) at press time
    delay_ms: u64,                // snapshot of manifest timing at press time
    interval_ms: u64,
    next_repeat: Option<Instant>, // None until the first tick schedules it
}
```

- **`handle_key_down`** (non-media branch): after `combo_down` succeeds, read
  `profile.repeats(key)` and `set.autorepeat()` once (while the read guard is held) and store them
  in the `HeldKey` with `next_repeat = None`. Media keys still tap via `press` and are never
  inserted — they never repeat.
- **New `pub fn tick(&mut self, now: Instant)`:** for each held key with `repeat == true` **and**
  `combo.key.is_some()`:
  - if `next_repeat` is `None` → set it to `now + delay_ms` (initial delay measured from the first
    tick after press — ≤ tick-interval slop, imperceptible);
  - else `while now >= next_repeat` → re-fire `self.injector.key_down(combo.key)` and advance
    `next_repeat += interval_ms`. (The `while` drains any backlog if a tick was late, but each
    re-fire advances by `interval_ms` so it self-limits.)
  - Skip keys with `repeat == false` or `combo.key == None` (modifier-only holds but never types).
- **`handle_key_up`** and **`release_held`** are unchanged in contract: removing / draining the
  entry ends its repeats. `combo_up` still lifts the key once (many key-downs, one key-up — matches
  a real keyboard).
- Snapshot rationale: reading timing/enable at press time avoids taking the `RwLock` every ~15ms;
  a mid-hold config change applying only on the next press is acceptable.

Re-fire reuses the existing `injector.key_down(&str)` — modifiers stay logically down from
`combo_down`, so `ctrl+c` repeats the `c` with ctrl held. **No injector changes.**

### `src/runtime.rs` and `src/monitor/mod.rs` (manual-verify)

- **GUI `consumer_loop`:** drop the `recv_timeout` from `50ms` to `~15ms` and call
  `dispatcher.tick(Instant::now())` on **both** the `Ok(event)` and `Timeout` branches (so repeats
  fire whether or not other events are arriving). The existing dry-run-toggle handling is
  unaffected (a shorter timeout only makes it more responsive).
- **Headless `run_headless`:** replace `for event in rx` with a loop on
  `rx.recv_timeout(Duration::from_millis(15))`: on `Ok(event)` handle it then `tick`; on `Timeout`
  `tick`; on `Disconnected` break and `release_held`. Preserves the existing exit-and-release
  behavior.
- When `held_keys` is empty, `tick` iterates an empty map — negligible cost at ~66 ticks/sec.

### `src/monitor/mod.rs` — Bindings tab (manual-verify)

- `render_binding_row` (shared by the G-key grid and the thumb section) gains a **repeat
  checkbox** to the right of the key text field, bound to a per-key entry in a
  `repeat_edits: HashMap<G13Key, bool>` buffer built on reload from `profile.repeats(key)`.
- **Save** writes both the `[keys]` bindings and the `[repeat]` flags (via `Profile::to_toml`).
  **Revert** reloads both buffers.
- A small caption near the section notes that repeat **timing** is set in `config.toml`
  (`[autorepeat]`), so it is discoverable without a Settings tab.
- **Monitor tab:** no change (a held key already lights in the grid; repeat is behavioral).

## Error handling

- Re-fire failures log a warning and continue (existing policy — a missed repeat is recoverable).
- Timing clamped at load (`interval_ms >= 1`) so config cannot cause a busy-spin.
- No `panic!`/`unwrap()` in the tick path.

## Testing

**Unit (TDD):**
- **Dispatcher `tick`** (deterministic via an injected `now: Instant`, using a base
  `Instant::now()` plus `Duration` offsets):
  - repeat-enabled key: `tick` before `delay` → no re-fire; at `delay` → exactly one re-fire; at
    `delay + interval` → a second. Re-fires recorded as `key_down(combo.key)`.
  - repeat-**disabled** key: no `tick` ever re-fires (holds only).
  - modifier-only combo with repeat enabled: no re-fire.
  - media key: tapped, never held, never repeats.
  - after `KeyUp` and after `release_held`: no further re-fires.
  - combo (`ctrl+c`) with repeat: re-fires the `c` only (modifiers stay held).
- **Config:** `[autorepeat]` parses `delay_ms`/`interval_ms`; absent → defaults (400/40);
  `interval_ms = 0` clamps to `1`. `[repeat]` parses per-key flags; `repeats()` defaults `false`;
  `to_toml` round-trip preserves enabled flags and omits disabled ones.

**Manual-verify (documented exception):** consumer-loop tick wiring (GUI + headless) and the GUI
checkbox.

**Hardware smoke test:** bind a key with repeat on, hold in Notepad → characters repeat after the
delay at the interval; bind another with repeat off → holds but types once; a repeating key does
not stick after release; a game sees a held key normally.

## Out of scope (future)
- GUI editor for the global `[autorepeat]` timing (defer to the planned Settings tab; edited in
  `config.toml` by hand for now, picked up on hot-reload).
- Per-binding timing overrides (timing is global only).
- Reading the user's actual Windows typematic settings (`SPI_GETKEYBOARDDELAY/SPEED`) as the
  default — would leak a Win32 call into the config layer; fixed defaults are simpler.
- Auto-repeat for joystick WASD directions (movement, not typing — excluded by design).
