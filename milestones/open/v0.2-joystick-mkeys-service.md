# v0.2 — joystick, M-key profiles, Windows Service

- **Status:** open
- **Target:** v0.2
- **Updated:** 2026-06-30

## Goal
Make the driver usable for gaming and run unattended: joystick input, profile switching
via the M-keys, and installation as a background Windows Service.

## Sub-projects
This milestone is built as three independent spec→plan→build cycles:
1. **Joystick → WASD/movement** — ✅ DONE (see below)
2. **M-key decode + profile switching** — ✅ DONE (see below)
3. Windows Service — open

## Tasks
- [x] Decode joystick X/Y (report **bytes 1–2**, NOT 5–6 — corrected on hardware) and the
      joystick button (**byte 7 bit3**, NOT byte 4).
- [x] Map joystick -> WASD movement (config-driven, per-axis 8-way thresholding). *Mouse
      mode is designed into the `[joystick] mode` seam but not yet implemented.*
- [x] Decode M-keys (M1/M2/M3 = **byte 6 bits 5–7**, MR = **byte 7 bit0** — corrected on
      hardware, NOT byte 4).
- [x] Profile switching: M-keys select among **profiles** (each a full binding-set file in
      `profiles/`; `config.toml` is a manifest naming M1/M2/M3). MR reserved.
- [ ] Run as a Windows Service (install/uninstall), promoted from console app.

## Sub-project 2 outcome (2026-07-04) — M-key profiles
Hardware-verified. Spec: `docs/superpowers/specs/2026-07-04-mkey-profiles-design.md`;
plan: `docs/superpowers/plans/2026-07-04-mkey-profiles.md`. Built via subagent-driven
development on `feat/mkey-profiles` (6 tasks, each reviewed).
- `ReportParser` decodes M-keys (bytes 6/7); `MKey`/`MKeyDown`/`MKeyUp` events.
- `Config` renamed to `Profile`; new `Manifest` + `ProfileSet` (loaded profiles + active
  slot) replace it behind `Arc<RwLock>`. `config.toml` is a manifest → `profiles/` files.
- Dispatcher reads the active profile; `MKeyDown(M1/M2/M3)` releases held keys then switches;
  MR reserved. Runtime watches config + profiles dir and reloads preserving the active slot.
- GUI: header shows the active profile, Monitor tab has an M-key indicator, Profiles tab
  lists/switches profiles (click or M-key). Legacy bare-`[keys]` config still loads as M1.
- Verified on hardware: M1/M2/M3 switch profiles (header + injected bindings change), no
  stuck key on switch, click-to-switch works, hot-reload works.

### Follow-ups
- Assigning files to slots + editing bindings in the GUI (New/Rename/Delete still inert).
- M-key **LEDs** (needs the output protocol); **MR** behavior.
- A profile file that *fails to load* shows in the GUI as `(unassigned)` (same as unset) —

## Acceptance
Joystick drives movement in a test app; M-keys swap active profile live; service installs,
auto-starts, and survives logout.

## Sub-project 1 outcome (2026-06-30) — Joystick → WASD
Built via subagent-driven development on branch `feat/joystick-wasd` (6 tasks, TDD, each
reviewed). **Hardware-verified on the real G13.** Spec: `docs/superpowers/specs/2026-06-30-joystick-wasd-movement-design.md`; plan: `docs/superpowers/plans/2026-06-30-joystick-wasd-movement.md`.

- `ReportParser` emits `G13Event::JoystickMove { x, y }` from bytes 1,2.
- `JoystickMapper` (`src/joystick.rs`): per-axis thresholding (center 127, configurable
  deadzone), 8-way (diagonals hold two keys), cross-center swaps without stuck keys.
- `KeyInjector` gained `key_down`/`key_up` (true hold); dispatcher routes `JoystickMove`
  live from config (hot-reload works).
- Hardware test confirmed: hold W/A/S/D on deflection, release on recenter, diagonals,
  live rebind (`up = "w"` → `"i"`), no `SendInput` warnings, no stuck keys.

### Known limitation / follow-up
- **Stuck key on Ctrl+C:** `release_held()` (which lifts held keys) only runs when the
  dispatch loop exits — i.e. on USB read error / device unplug (channel close). There is
  **no console Ctrl+C handler**, so Ctrl+C / force-kill terminates the process before
  cleanup, and a held movement key can stay down in the OS. Fix: add a Windows console
  control handler (`SetConsoleCtrlHandler`) or the `ctrlc` crate that triggers
  `release_held` on SIGINT. Tracked for a small follow-up (also relevant to the Windows
  Service sub-project, which needs graceful stop handling anyway).
- Holding emits a single synthetic `key_down` (no auto-repeat) — correct for games that
  read key *state*; means text fields show one character per push, not repeats.
